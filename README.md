# JpaShop
JpaShopプロジェクトのフェッチ戦略として、LAZY（遅延ローディング）を使用しました。 理由は、EAGER（即時ローディング）の場合、エンティティテーブルにアクセスするたびに関連するすべてのフィールドのクエリが実行され、リソースの無駄が発生するのに対し、LAZYローディングはまずProxyオブジェクトを設定し、実際のフィールドにアクセスした際にそのデータを初期化するため、EAGERローディングより性能が有利です。しかし、LAZYローディングによるクエリのn回呼び出し問題は、fetch joinで1回のクエリに必要なフィールドを一度に取得し、ネットワークトラフィックを削減して性能を改善しました。 また、QueryDSL ライブラリを使用して動的クエリを処理し、基本的な CRUD は Spring Data JPA ライブラリを活用しました。

JpaShop 프로젝트의 페치 전략으로, LAZY(지연 로딩)을 사용했습니다. 
이유는 EAGER(즉시 로딩)인 경우, 엔티티 테이블에 접근할 때마다 관련 모든 필드의 쿼리가 실행되어, 리소스 낭비가 발생하는 반면,
LAZY 로딩은 우선 Proxy객체를 설정하고, 실제 필드에 접근했을때, 해당 데이터를 초기화하여, EAGER 로딩보다 성능에 유리합니다만,
LAZY 로딩에 의한 쿼리 n회 호출 issue는, fetch join으로 1회의 쿼리에 필요한 필드를 한 번에 가져와,
네트워크 트래픽을 줄여, 성능을 개선했습니다. 
또한 QueryDSL 라이브러리를 사용해 동적 쿼리를 처리하고, 기본적인, CRUD는, 
Spring Data JPA 라이브러리를 활용하였습니다.

JpaShop 프로젝트의 OrderController에서는 엔티티 객체를 직접 뷰로 렌더링하고 있습니다. 한편
OrderApiControlle 에서는 생성자를 통해 엔티티를 DTO 객체로 변환한 뒤 @RestController를 통해 JSON 으로 반환되고, 
반환된 JSON 데이터는 Postman에서 확인했습니다. 
외부 공개용 API에서 엔티티를 DTO로 변환하는 이유는, 엔티티 구조를 숨기고 필요한 컬럼만 추출하여, 
API 사양별로 유연하게 대응하기 위함입니다. 또한, 양방향 연관관계에서 엔티티를 직접 반환하면, 
JSON 생성 시 순환 참조에 의한 무한 루프가 발생할 우려가 있지만, DTO를 개입시킴으로, 이를 방지할 수 있습니다.
